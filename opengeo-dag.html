Part II
Cloud-Native
Geospatial
A community-built set of formats that eliminate the need for a server — and scale through the power of cloud storage.
COG
Cloud-Optimized GeoTIFF
Stream raster data for visualization and analysis with just HTTP range requests — no tile server needed.
Still a valid GeoTIFF, so every legacy tool reads it exactly as before.
Zarr
Cloud-Native Array Format
Cloud-native format for multi-dimensional data cubes — think HDF5 or NetCDF, built for the cloud.
Not backwards compatible like COG/COPC, but getting great traction in climate and earth science.
COPC
Cloud-Optimized Point Cloud
Stream point clouds for visualization and analysis — the COG equivalent for LiDAR data like AHN.
Still a valid .laz file, so it works with all existing LiDAR tools without conversion.
PMTiles
Single-File Map Tile Archive
One file on object storage serves all vector and raster map tiles via HTTP range requests — no tile server.
Deploy a fully interactive web map with zero infrastructure — just a file on S3.
GeoParquet
Columnar Geospatial Format
Query spatial data directly from DuckDB, BigQuery, or Pandas — no server, just SQL on a file in S3.
The format that makes AI integration with spatial data completely natural.
STAC
SpatioTemporal Asset Catalog
Simple JSON format that adds rich metadata to any cloud-native geospatial asset — making it searchable and AI-readable.
STAC API 1.0 enables search of all records using the standard OGC Features API.

Sovereignty

On European Storage
This architecture requires only S3-compatible object storage — the most commoditized layer of cloud. There are now multiple European providers. That's all Portolan needs.
Cloud-Native ≠ Big Cloud
People hear "cloud-native" and assume AWS or Azure. Portolan runs on MinIO, Cloudflare R2, Hetzner, or any S3-compatible storage — including on-premise.
Open Governance
Portolan is community-governed. No single organization controls it. Apache 2.0 license. Open decision-making from day one.
